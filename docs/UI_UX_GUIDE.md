# Sendora UI/UX 指南(功能定位篇)

> 本文件定義每個畫面「**是什麼、給誰用、解決什麼問題、主要動作是什麼**」。
> 技術實作見 `IMPLEMENTATION_GUIDE.md`(那份講怎麼做,這份講長什麼樣、為什麼存在)。
> 範圍與 Milestones 對齊實作指南的 M1–M5。

---

## 1. 產品定位

**一句話**:讓小型電商團隊把客戶資料倒進來、切出分群、發出行銷信、看見成效——四步閉環,不需要工程師。

使用者角色(影響每頁的權限與顯示):

| 角色 | 是誰 | 最常做的事 |
|---|---|---|
| Owner | 開團隊的人(老闆 / 負責人) | 看儀表板、管成員、管發送額度 |
| Member | 行銷人員 | 匯入、分群、建活動、發送 |

只有兩種角色:Owner 多出來的只有「管人、刪團隊、調額度」,其餘操作兩者相同。
設計上的含義:**主要使用者是行銷人員,不是工程師**。所有功能都要能用「選擇 + 填表」完成,
任何地方都不出現 SQL、JSON、API 字眼。

## 2. 整體版面與導航

```
┌──────────────────────────────────────────────────┐
│ Topbar:Logo | 團隊切換器 ▾   本月發送 1,234/5,000 | 頭像 ▾ │
├──────────┬───────────────────────────────────────┤
│ Sidebar  │                                       │
│ 📊 儀表板 │                                       │
│ 👥 客戶   │          主內容區                      │
│ 🏷 分群   │     (列表 / 表單 / 圖表)              │
│ ✉️ 活動   │                                       │
│ 📥 匯入   │                                       │
│ ⚙️ 設定   │                                       │
└──────────┴───────────────────────────────────────┘
```

- **Sidebar 只放六項**。行銷平台的通病是功能一多導航就深、使用者就迷路;
  功能再多也先收進這六個入口的二級頁。
- **團隊切換器在 Topbar**:多租戶的核心 UX,切換後整站 context 跟著換,
  URL 永遠帶 team slug(`/t/:slug/...`),書籤、分享連結都不會跑錯團隊。
- **本月發送用量常駐 Topbar**(`已發送 / 額度`):額度是發送前最常確認的數字,
  快用完時變黃色提醒。
- Desktop-first:行銷人員在辦公室用電腦操作,手機只保證「看儀表板」可讀,不為手機做操作流程。

## 3. 頁面清單與功能定位

### M1 — 帳號與團隊

| 頁面 | 定位 | 主要動作 |
|---|---|---|
| 註冊 / 登入 | 標準表單,不放行銷文案 | 進入產品 |
| 建立團隊(Onboarding) | 註冊後強制第一步:取團隊名稱與 slug | 建立 → 直接進儀表板 |
| 設定 > 成員 | Owner 管人:邀請(email)、移除 | 邀請成員 |
| 設定 > 團隊 | 改名、本月用量與額度顯示、危險區(刪除團隊) | — |

**Onboarding 的功能定位**:新團隊進來是「空的」,儀表板沒資料毫無意義。
所以建完團隊後的儀表板要呈現**引導式空狀態**(見 §5),把使用者推向「匯入第一批客戶」。

**額度的呈現定位**:不做點數、不做流水帳。Topbar 常駐用量 + 設定頁一行說明
(「本月已發送 1,234 / 5,000 封,每月 1 日重置」),要調額度找管理者——透明但不隆重。

### M2 — 客戶與匯入

| 頁面 | 定位 | 主要動作 |
|---|---|---|
| 客戶列表 | 整個產品的「資料地基」。可搜尋(email/姓名)、可按標籤篩、可分頁 | 搜尋、點進詳情 |
| 客戶詳情 | 單一客戶的 360 度視角:基本資料、標籤、訂單記錄、收信記錄、退訂 / 退信狀態徽章 | 貼 / 撕標籤 |
| 匯入首頁 | 上傳 CSV 的入口 + 歷史匯入紀錄列表(狀態:處理中 / 完成 / 部分失敗) | 上傳檔案 |
| 匯入詳情 | 單次匯入的結果報告:成功 N 筆、失敗 M 筆、每筆失敗的原因 | 下載失敗清單 |

**匯入的 UX 立場**:這是新手的第一個關卡,也是最容易挫折的地方。三個原則:

1. 上傳前就把格式講清楚:頁面上放欄位範例表 + 「下載範本 CSV」按鈕。
2. 失敗不神秘:每一筆失敗列都顯示人話原因(「第 12 列:email 格式錯誤」),不丟整批錯誤。
3. 進度可見:處理中的匯入用 Turbo Frame 自動刷新狀態,使用者不用按 F5。

### M3 — 標籤與分群

| 頁面 | 定位 | 主要動作 |
|---|---|---|
| 標籤管理(客戶頁的子功能) | 建 / 改 / 刪標籤,顯示每個標籤的客戶數 | 建標籤 |
| 分群列表 | 已儲存的 Audience 清單,每列顯示名稱、條件摘要、預估人數 | 建分群、點進編輯 |
| 分群編輯 | **整個產品的樞紐頁**(見下) | 儲存、預覽人數 |

**分群編輯頁的功能定位**:這是 Sendora 的核心價值所在——「不會寫 SQL 的人切出精準客群」。

- 條件列 UI:每列 = 「欄位 ▾ + 運算 ▾ + 值」,可加可刪,列與列之間是 AND。
  第一版不做 OR / 巢狀群組——複雜條件樹是進階需求,先求 80% 場景。
- 可選欄位收斂為五類:標籤、消費金額、訂單數、最近購買日、建立日期。
- **「預覽人數」是這頁的靈魂**:每次條件變動,按一下立即顯示「符合 1,234 人」。
  即時回饋讓使用者敢於嘗試,這是和「丟需求給工程師等三天」的根本差異。
- 分群是**動態的**:儲存的是條件不是名單,發送當下才解析。頁面上要明示這點
  (「此分群會在每次使用時重新計算」),避免使用者誤解人數固定。

### M4 — 行銷活動

| 頁面 | 定位 | 主要動作 |
|---|---|---|
| 活動列表 | 所有 Campaign,狀態徽章(草稿 / 發送中 / 已完成 / 失敗)+ 「複製」動作 | 建活動 |
| 活動編輯(草稿) | 兩段式表單:①對象(選分群)②內容(主旨 + 內文) | 前往發送確認 |
| 發送確認 | **唯一的全螢幕確認頁**:對象人數、本月剩餘額度夠不夠,一目了然 | 確認發送 |
| 活動成效 | 發送後的結果頁:寄達 / 失敗數字,失敗清單 | 下載成效 CSV |
| 退訂頁(公開) | 不在導航、不需登入:一句話 + 確認鈕,按完即生效,不做挽留問卷 | 確認退訂 |

**發送確認頁的功能定位**:發信是**不可逆**的動作,是全產品唯一需要「隆重確認」的地方。
確認頁明列:「將發送給 1,234 位客戶,本月剩餘額度 3,766 封」,並註明
「已自動排除退訂與退信無效的客戶」——人數和分群預覽可能不同,要先講。
額度不足時,確認按鈕反灰並說明原因(「超出本月額度,請聯絡管理者調整」)。
其他所有刪除類動作用 modal confirm 即可,不要到處都隆重,隆重就失去意義。

**只有立即發送**:第一版沒有排程。按下確認 = 馬上進發送佇列,狀態徽章從「發送中」
自動更新到「已完成」。排程是之後的擴充,不影響這頁的結構。

**成效的取捨**:第一版只看「寄達 / 失敗」,不做開信率 / 點擊率 / 漏斗——
那需要追蹤 pixel 與轉址,屬於後續擴充;對「這批信發出去了沒、誰失敗了」這個
核心問題,寄達 / 失敗已經回答完了。

**內容編輯的取捨**:第一版用「主旨 + 簡單富文本(或純文字 + 變數插入,如 {{name}})」。
之後要升級,可嵌入 Unlayer 之類的第三方拖拉式編輯器,不影響資料模型。

### M5 — 儀表板與報表

| 頁面 | 定位 | 主要動作 |
|---|---|---|
| 儀表板(首頁) | 老闆視角的「這個月生意如何」:4 張 KPI 卡 + 2 張趨勢圖 + 近期活動結果表 | 切日期區間 |
| 報表匯出 | 不是獨立頁,是各列表頁右上角的「匯出 CSV」按鈕 | 下載 |

**儀表板的功能定位**:它是登入後的第一頁,回答三個問題就好——

1. 營收與訂單趨勢如何?(折線圖,日粒度)
2. 客戶在成長嗎?(新客 / 活躍客 KPI 卡)
3. 上次發的活動發成功了嗎?(近 5 筆 campaign 的寄達 / 失敗表格)

明確**不做**:自訂報表、拖拉式 dashboard、20 種圖表類型。深度數據分析是另一個
產品級別的功能,Sendora 的儀表板定位是「健康檢查」不是「分析工具」。

## 4. 三條核心使用者旅程

### 旅程 A:首次上手(目標:30 分鐘內發出第一封信)

```
註冊 → 建團隊 → 儀表板空狀態引導 → 下載 CSV 範本 → 匯入客戶
→ 匯入完成 → 建第一個分群(預覽人數!) → 建活動 → 確認頁 → 發送 → 看結果
```

每一步的「下一步」都由介面主動指出(空狀態 CTA),不靠文件。

### 旅程 B:日常發送(每週重複,目標:5 分鐘完成)

```
登入 → 活動列表 → 複製上次活動 → 改主旨內文 → 換分群 → 確認發送 → 完成
```

含義:活動列表要有「複製」動作——重複行銷是常態,別讓使用者每次從零建。

### 旅程 C:月初檢視(Owner,目標:1 分鐘看完)

```
登入 → 儀表板(預設本月)→ 切上月對照 → 看活動結果表 → 登出
```

含義:儀表板預設區間要是「本月 vs 上月」,不要讓老闆自己調日期。

## 5. 空狀態(Empty States)規範

多租戶 SaaS 的新團隊全是空頁面,空狀態就是 onboarding。每個列表頁的空狀態必含:
一句話說明這功能幹嘛 + 一個主 CTA。

| 頁面 | 空狀態文案方向 | CTA |
|---|---|---|
| 儀表板 | 「還沒有資料。從匯入第一批客戶開始」 | 前往匯入 |
| 客戶列表 | 「匯入 CSV 或手動新增第一位客戶」 | 匯入 / 新增 |
| 分群列表 | 「建立第一個分群,精準鎖定你的客戶」 | 建立分群(需先有客戶,否則導去匯入) |
| 活動列表 | 「對某個分群發出第一封行銷信」 | 建立活動(需先有分群,否則導去分群) |

注意依賴鏈:活動依賴分群、分群依賴客戶。空狀態的 CTA 要指向**前置條件**,不是死按鈕。

## 6. UI 模式約定(全站一致)

- **列表頁模式**:頁標題 + 右上主 CTA(「建立 X」)+ 篩選列 + 表格 + 分頁。
  全站所有列表長一樣,學一次到處會用。
- **表單頁模式**:單欄、由上而下、必填在前;錯誤訊息 inline 顯示在欄位下方
  (Rails 表單驗證的標準呈現),不用 toast 報錯。
- **Modal 的使用時機**:只用於「不需要 context 的小表單」(建標籤、改名、confirm)。
  分群編輯、活動編輯這種主工作流一律是完整頁面,不塞 modal。
- **即時回饋**:成功操作用一條 flash message(綠色,3 秒自動消失);
  背景處理中的東西(匯入、發送)用狀態徽章 + Turbo Frame 自動刷新,**全站不出現轉圈圈遮罩**。
- **危險動作**:刪除一律二次確認;「發送 Campaign」用專屬確認頁(§3 M4);
  刪除團隊要打字確認 slug。
- **數字格式**:金額帶千分位、百分比一位小數、日期 `YYYY-MM-DD`,全站統一。

## 7. 視覺風格

- **基調**:乾淨的 B2B 工具風——白底、灰階為主、一個品牌主色(建議靛藍系)、
  語意色只用於狀態(綠=成功、黃=處理中 / 額度將盡、紅=失敗)。
- **字體**:系統字型堆疊即可(中文環境下自訂 webfont 成本高收益低)。
- **密度**:表格行高偏緊湊(行銷人員要掃大量資料),表單偏寬鬆(填寫是低頻慎重操作)。
- **圖表**:Chart.js 預設樣式微調品牌色即可,折線圖 + 長條圖兩種就夠,拒絕圓餅圖氾濫。

## 8. 設計原則總結(取捨的依據)

1. **四步閉環優先**:匯入 → 分群 → 發送 → 看成效。任何功能若不在這條鏈上,先不做
   (點數計費、排程發送、開信追蹤就是這樣被排除的)。
2. **即時回饋換信心**:預覽人數、匯入進度、發送狀態——使用者隨時知道系統在幹嘛。
3. **隆重只給一件事**:發送(不可逆 + 吃額度)。其他操作越輕越好。
4. **空狀態即引導**:不做產品導覽(tour)、不寫使用手冊,介面自己會教。
5. **像個工具,不像個平台**:六個導航項、一種列表模式、一種表單模式。
   優勢是「小而清楚」,守住它。
